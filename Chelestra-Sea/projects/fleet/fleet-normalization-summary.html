The Problem

After consolidating from 31 to 20 agents, we discovered the fleet was running with inconsistent configs — different memory backends, different sandbox modes, different exec permissions, different compaction strategies. Some agents had GB-sized memory databases from unchecked session indexing. Others were burning tokens on Gemini embeddings. The "default root" config that should have been the source of truth wasn't being used by any running agent.

The fleet didn't have one memory strategy. It had several accidental ones.

The Solution: 7 Role-Based Profiles

Instead of making every agent equally powerful (expensive and risky), or manually tuning each of 20 agents (unmaintainable), we designed 7 profile types based on what agents actually do:

Key Design Decisions

Memory: QMD Fleet-Wide

All 20 agents moved to QMD local memory with local embeddings. Zero API cost for memory search. The old builtin memory system was using Gemini embeddings ($$$) and creating multi-GB sqlite databases from session indexing.

Sandbox: Tiered by Role

Not every agent needs full filesystem and network access. Art agents (Ciang, Trian) and documentation agents (Ramu) run in sandbox: all. Coders and orchestrators run in sandbox: non-main (sandbox subagents only). Hugh (Capital-Risk) is locked down with subagents: 1/1/1, restart: false, agentToAgent: false.

Session Memory: Selective

Not every agent needs to index conversation history. Architects and orchestrators get sessionMemory: true for cross-session recall. Standard workers and senior devs get sessionMemory: false — they work on current context, not history.

No Hidden Defaults

Every profile explicitly sets all 56 configuration fields. No relying on inherited defaults, no guessing what "unset" means, no accidental drift.
